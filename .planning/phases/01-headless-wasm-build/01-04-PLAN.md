    - Dockerfile (build environment definition)
    - docker-compose.yml (build service configuration)
  </read_first>
  <action>
    Set up the Emscripten build environment and run the full three-stage build pipeline. Follow this priority order for environment setup:

    **Priority 1 -- Docker (preferred, matches existing infrastructure):**
    1. Check if Docker daemon is running: `docker info`
    2. If running, build the Docker image: `docker-compose build blender-wasm-build`
    3. Execute the three-stage pipeline inside Docker:
       ```
       docker-compose run --rm blender-wasm-build bash -c \
         "./scripts/build-host-tools.sh && ./scripts/build-wasm-deps.sh && ./scripts/build-wasm.sh"
       ```

    **Priority 2 -- Local emsdk install (if Docker unavailable):**
    1. Clone emsdk: `git clone https://github.com/emscripten-core/emsdk.git /tmp/emsdk`
    2. Install and activate: `cd /tmp/emsdk && ./emsdk install 5.0.4 && ./emsdk activate 5.0.4`
    3. Source environment: `source /tmp/emsdk/emsdk_env.sh`
    4. Install brotli if missing: `brew install brotli` (macOS)
    5. Run each stage sequentially:
       - `./scripts/build-host-tools.sh`
       - `./scripts/build-wasm-deps.sh`
       - `./scripts/build-wasm.sh`

    **CRITICAL -- Iterative error fixing:**
    The build WILL encounter compilation errors on first attempt. This is expected for a 4.4M line codebase being cross-compiled. For EACH error:

    1. Read the FULL error output carefully -- identify the specific file, line, and error type
    2. Common error categories and fixes:
       - **Missing include paths:** Add paths to cmake/wasm_sources.cmake WASM_BLENDER_INCLUDE_DIRS
       - **Missing library links:** Add to cmake/emscripten_overrides.cmake or build-wasm.sh CMake flags
       - **API incompatibility with Emscripten:** Modify source/wasm_headless_main.cc to use alternative API calls
       - **Blender API changes:** Update function signatures or includes in wasm_headless_main.cc to match current Blender 4.x API
       - **Thread-unsafe code:** Add guards or disable offending features via WITH_* flags
       - **Missing preprocessor defines:** Add -D flags to emscripten_overrides.cmake WASM_COMPILE_FLAGS
       - **CMake configuration errors:** Fix the -C cache files or build-wasm.sh CMake invocation
       - **Emscripten-specific link issues:** Adjust WASM_LINK_FLAGS in emscripten_overrides.cmake
    3. Fix the specific issue in the appropriate file (NOT in Blender Mirror -- that is read-only)
    4. Re-run the failed stage
    5. Repeat until the build completes or a blocking issue is hit

    **If the build hits an issue that cannot be resolved automatically:**
    - Document the exact error, what was tried, and what the user needs to do
    - Create a file `build-wasm/BUILD_STATUS.md` documenting: the error, attempted fixes, environment requirements, and manual steps needed
    - Still attempt to get as far as possible in the build

    **Volume path awareness (Docker vs local):**
    - Docker: Blender source at /src/blender, build output at /src/build-wasm
    - Local: Blender source at "./Blender Mirror", build output at ./build-wasm
    - The existing scripts handle both via PROJECT_ROOT and BLENDER_SRC variables

    **Build-wasm.sh path fix (if needed):**
    The existing docker-compose.yml mounts "Blender Mirror" at /src/blender, but build-wasm.sh sets BLENDER_SRC="${PROJECT_ROOT}/Blender Mirror". Inside Docker the PROJECT_ROOT is /src, so BLENDER_SRC resolves to "/src/Blender Mirror" which does NOT match the mount at /src/blender. If building inside Docker, either:
    - Update docker-compose.yml to mount at "/src/Blender Mirror" (matching the script), OR
    - Add a symlink inside the container: `ln -s /src/blender "/src/Blender Mirror"`, OR
    - Set BLENDER_SRC explicitly before running the script
